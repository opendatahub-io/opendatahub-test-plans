# TC-E2E-002: Session recovery during coding task

**Objective**: Validate that a multi-step Codex CLI coding task
survives a network interruption and resumes with full context from
PostgreSQL session state.

**Preconditions**:

- Same environment as TC-E2E-001 (sample repo, Codex CLI,
  OGX, vLLM, PostgreSQL)
- Network disruption tooling available (iptables rules or
  `oc delete pod` capability)
- Session recovery requires same `session_id` on reconnect

**Test Steps**:

1. Start a coding task via Codex CLI with
   `OPENAI_BASE_URL=http://llamastack-svc:8321/v1`
2. Complete 3 tool-call turns successfully (e.g., `bash(pytest)` ->
   `read_file(test.py)` -> `grep_search(error)`)
3. Record the `session_id` from the request headers or PostgreSQL
4. Query PostgreSQL to verify turn 3 exists for the recorded
   `session_id` before disruption
5. Simulate network disconnect by killing the OGX pod:
   `oc delete pod <ogx-pod> -n <namespace>`
6. Wait for the pod to restart and become ready with the configured
   rollout timeout:
   `oc wait pod -l app=llamastack --for=condition=Ready -n <namespace> --timeout=60s`
7. Query PostgreSQL again and verify the pre-disruption turn count is
   unchanged
8. Resume Codex CLI with the same `session_id`
9. Send a prompt referencing prior context: "Based on the error you
   found earlier, fix the source file"
10. Verify the model references tool results from turns 1-3 and query
    the tool invocation audit path to confirm turns 1-3 were not
    re-executed after restart

**Expected Results**:

- OGX pod becomes Ready within the configured 60-second wait timeout
- Codex CLI reconnects using the same `session_id`
- Model correctly references the pytest output from turn 1 and
  the file content from turn 2 without re-executing those tools
- Tool invocation audit records show no duplicate invocations for turns
  1-3 after reconnect
- Remaining tool calls (e.g., `edit_file`, `bash(pytest)`)
  complete successfully
- PostgreSQL session record shows continuous turn numbering
  across the disconnect

**Validation**:

- Before disruption, `psql -c "SELECT count(*) FROM session_turns WHERE
  session_id = '<session_id>'"` records the expected pre-disruption turn
  count
- After pod recovery and before reconnect, the same query returns the
  same count
- `psql -c "SELECT count(*) FROM session_turns WHERE
  session_id = '<session_id>'"` shows all turns (pre- and
  post-disconnect)
- No duplicate `turn_number` values exist for the session
- Tool invocation audit records show that pre-disruption tool calls were
  not repeated after reconnect
