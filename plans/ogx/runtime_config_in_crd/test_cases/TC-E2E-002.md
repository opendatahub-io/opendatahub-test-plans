# TC-E2E-002: overrideConfig — ConfigMap-based config to running server

**Objective**: Verify that an OGXServer CR with `spec.overrideConfig` referencing a user-provided
ConfigMap uses the custom config.yaml instead of generating one from spec fields, and the server
starts successfully.

**Preconditions**:

- OpenShift cluster with OGX operator installed
- `oc` CLI authenticated with cluster-admin privileges
- A namespace for the test (e.g., `e2e-override-config`)

**Test Steps**:

1. Create the test namespace:

   ```bash
   oc new-project e2e-override-config
   ```

2. Create a ConfigMap with a custom config.yaml:

   ```bash
   oc apply -f - <<EOF
   apiVersion: v1
   kind: ConfigMap
   metadata:
     name: ogx-custom-config
     namespace: e2e-override-config
   data:
     config.yaml: |
       version: '2'
       apis:
         - inference
         - safety
       providers:
         inference:
           - provider_id: stub-inference
             provider_type: inline::meta-reference
             config:
               model: "meta-llama/Llama-3.2-3B-Instruct"
         safety:
           - provider_id: stub-safety
             provider_type: inline::llama-guard
             config:
               model: "meta-llama/Llama-Guard-3-1B"
   EOF
   ```

3. Apply the OGXServer CR with `spec.overrideConfig` pointing to the custom ConfigMap:

   ```bash
   oc apply -f - <<EOF
   apiVersion: ogx.io/v1beta1
   kind: OGXServer
   metadata:
     name: ogx-override-test
     namespace: e2e-override-config
   spec:
     replicas: 1
     distribution:
       name: rh-dev
     overrideConfig:
       configMapRef:
         name: ogx-custom-config
         key: config.yaml
   EOF
   ```

4. Wait for the pod to become Ready:

   ```bash
   oc wait --for=condition=Ready pod -l app=ogx-override-test -n e2e-override-config --timeout=180s
   ```

5. Verify the Deployment volume mount references the user-provided ConfigMap (not a generated one):

   ```bash
   oc get deployment ogx-override-test -n e2e-override-config -o jsonpath='{.spec.template.spec.volumes[*].configMap.name}'
   ```

6. Verify the mounted config inside the pod matches the custom config.yaml:

   ```bash
   OGX_POD=$(oc get pods -l app=ogx-override-test -n e2e-override-config -o jsonpath='{.items[0].metadata.name}')
   oc exec $OGX_POD -n e2e-override-config -- cat /config/config.yaml
   ```

7. Verify the server health endpoint responds:

   ```bash
   oc exec $OGX_POD -n e2e-override-config -- curl -s http://localhost:8321/health
   ```

8. Verify no operator-generated ConfigMap was created (only the user-provided one):

   ```bash
   oc get configmaps -n e2e-override-config -o name
   ```

**Expected Results**:

- The user-provided ConfigMap `ogx-custom-config` is created in the namespace
- The OGXServer operator uses the custom ConfigMap directly instead of generating a new one from
  spec fields
- The Deployment's volume mount references `ogx-custom-config`, not a hash-suffixed generated
  ConfigMap
- The config.yaml inside the running pod matches the content provided in the custom ConfigMap
  exactly
- The health endpoint at `http://localhost:8321/health` returns a successful response
- No additional operator-generated ConfigMap with a content hash suffix exists in the namespace
