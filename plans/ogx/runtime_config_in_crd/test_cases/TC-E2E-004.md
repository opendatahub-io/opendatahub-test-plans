# TC-E2E-004: Legacy adoption journey

**Objective**: Verify that an OGXServer CR can adopt pre-existing legacy LLSD resources
(PVC, Service) via adoption annotations, that rolling updates preserve adopted PVCs, and that
deleting the CR orphans the adopted PVC instead of deleting it.

**Preconditions**:

- OpenShift cluster with OGX operator installed
- `oc` CLI authenticated with cluster-admin privileges
- A namespace with pre-existing legacy LLSD PVC and Service
  (simulating a 3.4-to-3.5 migration scenario)

**Test Steps**:

1. Create the test namespace and simulate legacy LLSD resources:

   ```bash
   oc new-project e2e-legacy-adoption

   # Create a legacy PVC (simulating LLSD storage)
   oc apply -f - <<EOF
   apiVersion: v1
   kind: PersistentVolumeClaim
   metadata:
     name: llsd-data
     namespace: e2e-legacy-adoption
     labels:
       app: llama-stack-distribution
   spec:
     accessModes:
       - ReadWriteOnce
     resources:
       requests:
         storage: 5Gi
   EOF

   # Create a legacy Service (simulating LLSD service)
   oc apply -f - <<EOF
   apiVersion: v1
   kind: Service
   metadata:
     name: llsd-service
     namespace: e2e-legacy-adoption
     labels:
       app: llama-stack-distribution
   spec:
     selector:
       app: llama-stack-distribution
     ports:
       - port: 8321
         targetPort: 8321
   EOF
   ```

2. Verify the legacy resources exist and are not owned by any CR:

   ```bash
   oc get pvc llsd-data -n e2e-legacy-adoption -o jsonpath='{.metadata.ownerReferences}'
   oc get svc llsd-service -n e2e-legacy-adoption -o jsonpath='{.metadata.ownerReferences}'
   ```

3. Apply an OGXServer CR with adoption annotations referencing the legacy resources:

   ```bash
   oc apply -f - <<EOF
   apiVersion: ogx.io/v1beta1
   kind: OGXServer
   metadata:
     name: ogx-adopted
     namespace: e2e-legacy-adoption
     annotations:
       ogx.io/adopt-pvc: "llsd-data"
       ogx.io/adopt-service: "llsd-service"
   spec:
     replicas: 1
     distribution:
       name: rh-dev
     workload:
       storage:
         size: 5Gi
   EOF
   ```

4. Wait for reconciliation:

   ```bash
   oc wait --for=condition=Ready pod -l app=ogx-adopted -n e2e-legacy-adoption --timeout=180s
   ```

5. Verify the legacy resources are now adopted (owned by the OGXServer CR):

   ```bash
   echo "=== PVC owner references ==="
   oc get pvc llsd-data -n e2e-legacy-adoption -o jsonpath='{.metadata.ownerReferences}' | python3 -m json.tool

   echo "=== Service owner references ==="
   oc get svc llsd-service -n e2e-legacy-adoption -o jsonpath='{.metadata.ownerReferences}' | python3 -m json.tool
   ```

6. Verify the pod is using the adopted PVC:

   ```bash
   oc get deployment ogx-adopted -n e2e-legacy-adoption -o jsonpath='{.spec.template.spec.volumes[*].persistentVolumeClaim.claimName}'
   ```

7. Trigger a rolling update by modifying the CR config:

   ```bash
   oc patch ogxserver ogx-adopted -n e2e-legacy-adoption --type=merge -p '
   spec:
     providers:
       inference:
         - provider_id: updated-provider
           provider_type: remote::vllm
           config:
             url: "https://updated-endpoint/v1"
   '
   ```

8. Wait for the rolling update to complete:

   ```bash
   oc rollout status deployment/ogx-adopted -n e2e-legacy-adoption --timeout=120s
   ```

9. Verify the adopted PVC still exists and is still bound after the rolling update:

   ```bash
   oc get pvc llsd-data -n e2e-legacy-adoption -o jsonpath='{.status.phase}'
   ```

10. Verify the new pod still references the adopted PVC:

    ```bash
    oc get deployment ogx-adopted -n e2e-legacy-adoption -o jsonpath='{.spec.template.spec.volumes[*].persistentVolumeClaim.claimName}'
    ```

11. Delete the OGXServer CR:

    ```bash
    oc delete ogxserver ogx-adopted -n e2e-legacy-adoption
    ```

12. Verify the adopted PVC is orphaned (still exists) but the Service is deleted:

    ```bash
    echo "=== PVC should still exist (orphaned) ==="
    oc get pvc llsd-data -n e2e-legacy-adoption

    echo "=== PVC owner references should be empty ==="
    oc get pvc llsd-data -n e2e-legacy-adoption -o jsonpath='{.metadata.ownerReferences}'

    echo "=== Deployment should be gone ==="
    oc get deployment ogx-adopted -n e2e-legacy-adoption 2>&1
    ```

**Expected Results**:

- Legacy PVC and Service exist without ownerReferences before CR creation
- After applying the OGXServer CR with adoption annotations, the legacy resources gain
  ownerReferences pointing to the CR
- The pod mounts the adopted PVC (`llsd-data`) for its storage
- After a rolling update triggered by config change, the adopted PVC is preserved
  (not deleted and recreated)
- The new pod after rolling update still uses the same adopted PVC
- After deleting the OGXServer CR, the adopted PVC is orphaned
  (remains in the namespace without ownerReferences) rather than being garbage-collected
- The Deployment and generated resources are cleaned up when the CR is deleted
