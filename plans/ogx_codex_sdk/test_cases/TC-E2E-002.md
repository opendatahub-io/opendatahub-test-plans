# TC-E2E-002: Session recovery during coding task

**Objective**: Validate that a multi-step Codex CLI coding task
survives a network interruption and resumes with full context from
PostgreSQL session state.

**Preconditions**:
- Same environment as TC-E2E-001 (sample repo, Codex CLI,
  LlamaStack, vLLM, PostgreSQL)
- Network disruption tooling available (iptables rules or
  `oc delete pod` capability)
- Session recovery requires same `session_id` on reconnect

**Test Steps**:
1. Start a coding task via Codex CLI with
   `OPENAI_BASE_URL=http://llamastack-svc:8321/v1`
2. Complete 3 tool-call turns successfully (e.g., `bash(pytest)` ->
   `read_file(test.py)` -> `grep_search(error)`)
3. Record the `session_id` from the request headers or PostgreSQL
4. Simulate network disconnect by killing the LlamaStack pod:
   `oc delete pod <llamastack-pod> -n <namespace>`
5. Wait for the pod to restart and become ready:
   `oc wait pod -l app=llamastack --for=condition=Ready -n <namespace>`
6. Resume Codex CLI with the same `session_id`
7. Send a prompt referencing prior context: "Based on the error you
   found earlier, fix the source file"
8. Verify the model references tool results from turns 1-3

**Expected Results**:
- LlamaStack pod restarts within 60 seconds
- Codex CLI reconnects using the same `session_id`
- Model correctly references the pytest output from turn 1 and
  the file content from turn 2 without re-executing those tools
- Remaining tool calls (e.g., `edit_file`, `bash(pytest)`)
  complete successfully
- PostgreSQL session record shows continuous turn numbering
  across the disconnect

**Validation**:
- `psql -c "SELECT count(*) FROM session_turns WHERE
  session_id = '<session_id>'"` shows all turns (pre- and
  post-disconnect)
- No duplicate `turn_number` values exist for the session
