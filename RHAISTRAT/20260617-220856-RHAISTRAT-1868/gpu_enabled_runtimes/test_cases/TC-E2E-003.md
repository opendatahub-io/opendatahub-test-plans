# TC-E2E-003: Side-by-side CPU and GPU deployment with cross-validation

**Objective**: Validate the end-to-end scenario of deploying the same
ResNet-50 ONNX model on both CPU (`mlserver-onnx`) and GPU
(`mlserver-cuda-runtime`) runtimes simultaneously, and
cross-validating inference results for numerical consistency.

**Preconditions**:
- OCP 4.20+ cluster with RHOAI 3.5 GA and NVIDIA GPU Operator 12.9+
- Both `mlserver-onnx` (CPU) and `mlserver-cuda-runtime` (GPU)
  ClusterServingRuntimes available
- HardwareProfile `nvidia-gpu-a100` (API group
  `infrastructure.opendatahub.io/v1`) available
- GPU nodes and CPU-only nodes available
- Same ResNet-50 ONNX model artifact in S3 storage

**Test Steps**:
1. Deploy InferenceService on CPU runtime:
   ```bash
   cat <<EOF | oc apply -f -
   apiVersion: serving.kserve.io/v1beta1
   kind: InferenceService
   metadata:
     name: resnet-cpu
     annotations:
       serving.kserve.io/deploymentMode: RawDeployment
   spec:
     predictor:
       model:
         modelFormat:
           name: onnx
           version: "1"
         runtime: mlserver-onnx
         storageUri: s3://models/resnet-50-onnx/
   EOF
   oc wait inferenceservice/resnet-cpu --for=condition=Ready \
     --timeout=300s
   ```
2. Deploy InferenceService on GPU runtime:
   ```bash
   cat <<EOF | oc apply -f -
   apiVersion: serving.kserve.io/v1beta1
   kind: InferenceService
   metadata:
     name: resnet-gpu
     annotations:
       serving.kserve.io/deploymentMode: RawDeployment
   spec:
     predictor:
       model:
         modelFormat:
           name: onnx
           version: "1"
         runtime: mlserver-cuda-runtime
         storageUri: s3://models/resnet-50-onnx/
   EOF
   oc wait inferenceservice/resnet-gpu --for=condition=Ready \
     --timeout=300s
   ```
3. Verify both InferenceServices are `Ready`:
   ```bash
   oc get inferenceservice resnet-cpu resnet-gpu \
     -o custom-columns="NAME:.metadata.name,READY:.status.conditions[?(@.type=='Ready')].status"
   ```
4. Send the same inference request to the CPU runtime:
   ```bash
   CPU_URL=$(oc get inferenceservice resnet-cpu \
     -o jsonpath='{.status.url}')
   CPU_RESP=$(curl -s -X POST \
     "${CPU_URL}/v2/models/resnet-cpu/infer" \
     -H "Content-Type: application/json" \
     -d @resnet-input.json)
   ```
5. Send the identical request to the GPU runtime:
   ```bash
   GPU_URL=$(oc get inferenceservice resnet-gpu \
     -o jsonpath='{.status.url}')
   GPU_RESP=$(curl -s -X POST \
     "${GPU_URL}/v2/models/resnet-gpu/infer" \
     -H "Content-Type: application/json" \
     -d @resnet-input.json)
   ```
6. Compare the top-1 predicted class between CPU and GPU responses.
7. Verify resource allocation for each pod:
   ```bash
   oc get pod -l serving.kserve.io/inferenceservice=resnet-gpu \
     -o jsonpath='{.items[0].spec.containers[?(@.name=="kserve-container")].resources.limits}'
   oc get pod -l serving.kserve.io/inferenceservice=resnet-cpu \
     -o jsonpath='{.items[0].spec.containers[?(@.name=="kserve-container")].resources.limits}'
   ```

**Expected Results**:
- Both InferenceServices reach `Ready` state simultaneously
- Both return HTTP `200` with valid KServe V2 inference output
- Top-1 predicted class is identical between CPU and GPU responses
- GPU pod has `nvidia.com/gpu` in resource limits; CPU pod does not
- Both services function independently without interfering with
  each other
